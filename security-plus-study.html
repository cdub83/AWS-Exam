Security Control Categories & Types
⚠ Exam Trap
Know the two axes: Category (Technical, Managerial, Operational, Physical) vs Type (Preventive, Detective, Corrective, Deterrent, Compensating, Directive). A camera is Physical + Detective. A policy is Managerial + Directive. A fence is Physical + Deterrent.

Control Types

  • Preventive — stops an incident (firewall, lock)
  • Detective — identifies an incident (IDS, camera, audit log)
  • Corrective — restores after incident (backup restore, patch)
  • Deterrent — discourages attack (warning sign, guard)
  • Compensating — alternative when primary can't be used
  • Directive — instructs behaviour (policy, procedure)

Control Categories

  • Technical — firewalls, encryption, IDS/IPS, MFA
  • Managerial — policies, risk assessments, awareness training
  • Operational — procedures, background checks, security guards
  • Physical — locks, fences, cameras, mantraps, badge readers
Cryptography Fundamentals

Symmetric Encryption

  • Same key encrypts & decrypts
  • AES — 128/192/256-bit; current standard
  • 3DES — legacy; being phased out
  • Fast; used for bulk data encryption
  • Key distribution problem — how to share key securely?

Asymmetric Encryption

  • Key pair: public (encrypt) + private (decrypt)
  • RSA — 2048/4096-bit; key exchange & signatures
  • ECC — smaller keys, same strength; efficient on mobile
  • Slower; used for key exchange & signatures
  • Encrypt with recipient's public key → only private key decrypts

Hashing

  • One-way; cannot be reversed
  • SHA-256 / SHA-3 — current standard
  • MD5 — broken; do not use for security
  • Same input → same hash (deterministic)
  • Used for: file integrity, password storage (with salting), digital signatures

Digital Signatures

  • Sign with sender's private key
  • Verify with sender's public key
  • Provides: authentication, integrity, non-repudiation
  • Does NOT provide confidentiality
  • Process: hash the data → encrypt hash with private key

Key Management

  • Key escrow — copy of key stored with trusted third party
  • Key stretching — PBKDF2, bcrypt, Argon2 (make weak passwords stronger)
  • Salting — random value added before hashing to prevent rainbow tables
  • HSM — hardware security module; tamper-resistant key storage

Obfuscation & Data Hiding

  • Steganography — hide data inside a carrier file (image, audio)
  • Tokenisation — replace sensitive value with random token; no math relationship
  • Data masking — obscure part of data for display (****1234)
  • Obfuscation — make code/data harder to understand
Authentication & Identity

MFA Factors

  • Something you know — password, PIN
  • Something you have — smart card, OTP token, phone
  • Something you are — biometrics (fingerprint, retina, voice)
  • Somewhere you are — geolocation, IP range
  • MFA = two or more different factors
  • TOTP (time-based OTP) — 30-sec code via authenticator app; more secure than SMS

Federation & SSO

  • SAML 2.0 — XML-based identity federation; IdP issues assertion to SP
  • OAuth 2.0 — authorisation framework (not authentication)
  • OIDC — authentication layer on top of OAuth 2.0
  • Kerberos — Windows domain SSO; uses tickets (TGT → TGS)
  • SSO: authenticate once, access many systems

Access Control Models

  • DAC — owner controls access (Windows NTFS permissions)
  • MAC — system enforces labels (Top Secret, Secret, etc.)
  • RBAC — access based on role (most common in enterprises)
  • ABAC — access based on attributes (time, location, device)
  • Rule-based AC — firewall rules; not the same as RBAC
PKI & Certificates
⚠ Exam Trap
Certificate pinning bypasses normal certificate chain validation — the app only trusts a specific cert or CA, not the system trust store. This prevents MitM even with a rogue trusted CA. OCSP is real-time revocation checking (better than CRL for time-sensitivity).

PKI Components

  • CA — Certificate Authority; issues & signs certs
  • Root CA — top of the trust hierarchy; often offline
  • Intermediate CA — issues end-entity certs; online
  • CRL — Certificate Revocation List; periodic download
  • OCSP — Online Certificate Status Protocol; real-time revocation check
  • CSR — Certificate Signing Request; sent to CA for signing

Certificate Types

  • DV — Domain Validated; proves domain control only
  • OV — Organisation Validated; includes org identity
  • EV — Extended Validation; highest assurance
  • Wildcard — *.domain.com; covers all subdomains
  • SAN — Subject Alternative Name; multiple domains per cert
  • Self-signed — signed by own private key; not trusted by default

Social Engineering & Phishing

Phishing Variants

  • Phishing — bulk email to random targets
  • Spear phishing — targeted; uses personal details
  • Whaling — targets C-suite executives (as victims)
  • Vishing — voice / phone-based
  • Smishing — SMS text messages
  • BEC — impersonates executive to request wire transfer

Other Social Engineering

  • Pretexting — fabricated scenario to obtain info
  • Baiting — leaving infected USB drives; promise of reward
  • Quid pro quo — offer help in exchange for info/access
  • Tailgating/Piggybacking — following through a secure door
  • Typosquatting — lookalike domain (micros0ft.com)
  • Watering hole — compromise a site the target visits

SE Principles (Why It Works)

  • Authority — impersonates boss/police/IT
  • Urgency — "act now or lose access"
  • Social proof — "everyone else has agreed"
  • Scarcity — "only 1 spot left"
  • Liking/familiarity — appears as a friend or colleague
  • Fear — "you'll be arrested if you don't…"
Malware Types

Common Malware

  • Virus — attaches to file; requires human action to spread
  • Worm — self-replicates across network; no host file needed
  • Trojan — disguised as legitimate software
  • Ransomware — encrypts files; demands payment
  • Spyware — steals data/keystrokes silently
  • Adware — unwanted ads; may include spyware

Advanced Malware

  • Rootkit — hides its own presence; modifies OS; hard to detect
  • RAT — Remote Access Trojan; full remote control
  • Keylogger — records keystrokes (passwords, PINs)
  • Logic bomb — triggers on a condition (date, login, file deletion)
  • Fileless malware — lives in memory; no disk footprint; evades AV
  • Botnet/zombie — compromised host under C2 control
Network & Password Attacks

Password Attacks

  • Brute force — try all combinations
  • Dictionary — wordlist of common passwords
  • Password spray — one common password against many accounts (avoids lockout)
  • Credential stuffing — leaked username+password pairs from other breaches
  • Rainbow table — precomputed hash lookups; defeated by salting
  • Kerberoasting — request SPN tickets; crack offline for service account passwords

Network Attacks

  • DDoS — volumetric (UDP flood), protocol (SYN flood), application (HTTP flood)
  • On-path (MitM) — intercepts and potentially modifies traffic
  • Session hijacking — steals authenticated session token
  • Replay attack — captures and retransmits valid auth data
  • ARP spoofing — poisons ARP cache to redirect LAN traffic
  • DNS poisoning — injects false DNS records into resolver cache

Application Attacks

  • SQL injection — malicious SQL in input fields
  • XSS — injects scripts into pages viewed by other users
  • CSRF — tricks victim's browser into making unintended requests
  • Buffer overflow — overwrite adjacent memory; can execute code
  • Path traversal — ../../etc/passwd to access restricted files
  • IDOR — insecure direct object reference; access unauthorised records
✓ Key Differentiator
Password spray vs credential stuffing vs brute force: Spray = 1 password, many accounts. Credential stuffing = known pairs from another breach. Brute force = all combinations against one account. Spray avoids lockout; stuffing relies on password reuse.

Cloud Security

Cloud Service Models

  • IaaS — you manage OS up; provider manages hardware & hypervisor
  • PaaS — you manage app & data; provider manages runtime & OS
  • SaaS — you manage data & users only; provider manages everything else
  • Shared responsibility model — security "of" vs "in" the cloud

Cloud Deployment Models

  • Public — multi-tenant; shared infrastructure
  • Private — single-tenant; dedicated resources
  • Hybrid — on-premises + public cloud integration
  • Community — shared among organisations with common interests
  • Multi-cloud — use multiple cloud providers

Cloud Security Tools

  • CASB — Cloud Access Security Broker; visibility & control over SaaS
  • CSPM — Cloud Security Posture Management; find misconfigurations
  • CWPP — Cloud Workload Protection Platform; runtime protection
  • SWG — Secure Web Gateway; filter outbound web traffic
Network Architecture Patterns

Segmentation & Isolation

  • DMZ — semi-trusted zone between internet and internal
  • Screened subnet — another name for DMZ
  • Microsegmentation — workload-level firewall policies; limits lateral movement
  • VLAN — logical network separation at Layer 2
  • Air gap — physical isolation; no network connection

Zero Trust & SASE

  • Zero Trust — "never trust, always verify"; no implicit trust from location
  • Key components: strong identity, device posture, least privilege, microsegmentation
  • SASE — SD-WAN + FWaaS + CASB + SWG + ZTNA in one cloud service
  • SASE eliminates hairpinning; security at the edge near users

Infrastructure as Code (IaC)

  • Terraform, CloudFormation, Ansible — define infra in code files
  • IaC vulnerability: hardcoded secrets in config files committed to repos
  • Mitigation: use secrets managers, env variables, vault references
  • Enables consistent, repeatable, auditable deployments
Data Security & Resilience

Data States

  • Data at rest — stored on disk/database; protect with encryption (AES-256)
  • Data in transit — moving over network; protect with TLS/HTTPS/VPN
  • Data in use — in memory/CPU; protect with enclaves, access controls

Data Classification

  • Public — no restrictions; freely shareable
  • Internal/Private — internal use only; some restrictions
  • Confidential — business-sensitive; need-to-know basis
  • Top Secret/Restricted — highest sensitivity; strictly controlled
  • PII — personally identifiable information
  • PHI — protected health information (HIPAA)

Backup & Recovery

  • 3-2-1 rule — 3 copies, 2 media types, 1 offsite/offline
  • RTO — Recovery Time Objective: max acceptable downtime
  • RPO — Recovery Point Objective: max acceptable data loss
  • MTD — Maximum Tolerable Downtime: hard limit before catastrophic
  • Immutable backup / air-gapped copy is key defence against ransomware

HA & Site Types

  • Hot site — fully operational duplicate; immediate failover
  • Warm site — partially configured; hours to activate
  • Cold site — empty facility; days/weeks to bring up
  • RAID 1 — mirror; RAID 5 — parity; RAID 10 — mirror+stripe
  • Geographic diversity — different power grid, flood zone, etc.
⚠ Exam Trap
RTO vs RPO vs MTD: RTO is how fast you must restore. RPO is how much data loss is acceptable. MTD is the absolute maximum — if exceeded, the business fails. MTD > RTO always.

Enterprise Security Tools

SIEM, SOAR, EDR, XDR

  • SIEM — aggregates & correlates logs; generates alerts; requires analyst
  • SOAR — executes automated playbooks in response to alerts (no human needed)
  • EDR — endpoint behavioural monitoring, detection, threat hunting, automated response
  • XDR — EDR + network + cloud + email in unified platform

Network Security

  • IDS — detects intrusions; passive; alerts only
  • IPS — inline; actively blocks intrusions
  • WAF — HTTP/HTTPS layer protection; OWASP Top 10
  • NAC — evaluates device posture before network admission; quarantines non-compliant
  • DNS filtering — blocks resolution of malicious domains; covers entire network
  • DLP — prevents sensitive data leaving the organisation

Endpoint Security

  • MDM — manages full device: enforce policies, remote wipe, push configs
  • MAM — manages specific apps only (BYOD-friendly)
  • HIDS — host-based IDS; monitors files, processes, registry
  • Application control / whitelisting — only approved software runs
  • Secure boot — UEFI verifies bootloader integrity before loading OS

Privileged Access

  • PAM — Privileged Access Management; vaults admin credentials
  • Just-in-time (JIT) access — elevate privilege only when needed; auto-expires
  • Least privilege — users get only the access they need
  • Separation of duties — no single person controls entire process
Vulnerability Management

CVSS Scores

  • 0.1–3.9 = Low
  • 4.0–6.9 = Medium
  • 7.0–8.9 = High
  • 9.0–10.0 = Critical
  • Base score = exploitability + impact
  • Use CVSS + asset criticality + exploitability (CISA KEV) to prioritise

Scanning Types

  • Credentialed scan — logs in; sees installed software, configs, patches
  • Non-credentialed — external view; simulates attacker without login
  • Active scan — sends packets to probe; may cause disruption
  • Passive — captures traffic; no probing; no disruption
  • SAST — static application security testing (source code)
  • DAST — dynamic application security testing (running app)

Pen Test Phases

  • 1. Planning/Scoping — rules of engagement, authorisation
  • 2. Reconnaissance — passive (OSINT) then active
  • 3. Scanning — identify open ports, services, vulnerabilities
  • 4. Exploitation — gain initial access
  • 5. Post-exploitation — lateral movement, privilege escalation
  • 6. Reporting — findings, remediation recommendations
Incident Response & Digital Forensics

IR Lifecycle (NIST)

  • 1. Preparation — IR plan, tools, training, contacts
  • 2. Detection & Analysis — identify and confirm the incident
  • 3. Containment — limit the damage (short-term + long-term)
  • 4. Eradication — remove malware, close access
  • 5. Recovery — restore systems; monitor for recurrence
  • 6. Lessons learned — post-incident review; improve defences

Order of Volatility

  • 1. CPU registers & cache — milliseconds
  • 2. RAM / memory — processes, keys, network sockets
  • 3. Active network connections
  • 4. Running processes
  • 5. Temp files / swap
  • 6. Hard drive image — least volatile; collect last

Forensics Principles

  • Chain of custody — documented evidence handling
  • Legal hold — suspend deletion for litigation; overrides retention policy
  • Write blockers — prevent altering original evidence
  • Hashing evidence — MD5/SHA hash proves integrity
  • Never work on originals — always use forensic copies
  • Document everything — timestamps, tools, findings
Threat Intelligence

Intelligence Sharing

  • STIX — Structured Threat Information eXpression; JSON format for IoCs, TTPs
  • TAXII — Trusted Automated eXchange of Indicator Information; HTTP transport for STIX
  • ISAC — Information Sharing & Analysis Centre; sector-specific threat sharing
  • IoC — Indicator of Compromise (IP, hash, domain, email)
  • MITRE ATT&CK — adversary TTP framework

Threat Hunting

  • Proactive, hypothesis-driven search for attacker activity
  • Looks beyond what automated alerts detect
  • Uses endpoint telemetry, network logs, threat intel
  • Effective against APTs using living-off-the-land (LotL) techniques
  • LotL: abuses built-in OS tools (PowerShell, WMI, certutil)
⚠ Exam Trap
IDS vs IPS: IDS is passive — it alerts but does not block. IPS is inline — it can drop/block packets. A SIEM is neither — it aggregates logs and correlates events. SOAR adds automation to the SIEM workflow. None of these replace each other.

Risk Management

Risk Terminology

  • Threat — potential cause of harm
  • Vulnerability — weakness that can be exploited
  • Risk = Threat × Vulnerability × Impact
  • Likelihood — probability of threat occurring
  • Impact — consequence if it occurs
  • Risk appetite — how much risk an org is willing to accept

Risk Treatment Options

  • Avoid — don't do the risky activity
  • Mitigate / Reduce — implement controls to lower probability or impact
  • Transfer — cyber insurance; third-party contract
  • Accept — acknowledge risk; no action; document decision
  • Residual risk — risk remaining after controls applied
  • Inherent risk — risk before controls

Risk Assessment Methods

  • Qualitative — Low/Medium/High; uses judgment & opinion
  • Quantitative — SLE, ARO, ALE; uses dollar values
  • SLE (Single Loss Expectancy) = Asset value × Exposure factor
  • ALE (Annual Loss Expectancy) = SLE × ARO (Annual Rate of Occurrence)

Audits, Assessments & Business Continuity

Assessment Types

  • Vulnerability assessment — automated scan for known CVEs; passive
  • Penetration test — simulated attack; active exploitation
  • Red team — adversary simulation; full attack lifecycle
  • Blue team — defenders during an exercise
  • Purple team — red + blue collaborate in real time
  • Attestation — formal signed declaration controls exist (SOC 2, ISO 27001)

Pen Test Scopes

  • White box — full info: source code, network diagrams, credentials
  • Gray box — partial info: some credentials, basic topology
  • Black box — no info; simulates external attacker
  • Tabletop exercise — discussion-based; no actual systems attacked

BIA & BCP/DRP

  • BIA — identifies critical functions, MTD, RTO, RPO, financial impact
  • BCP — Business Continuity Plan: keep operations running during disruption
  • DRP — Disaster Recovery Plan: restore IT systems after incident
  • BIA feeds BCP feeds DRP — in that order
  • Test plans: tabletop → walkthrough → simulation → full parallel test

Data Governance

  • Data retention policy — how long to keep data (normal schedule)
  • Legal hold — overrides retention; preserve all relevant data for litigation
  • PIA / DPIA — assess privacy risks before high-risk processing begins
  • Data sovereignty — data subject to laws of country where stored
  • Data minimisation — collect only what is necessary
Third-Party & Change Management

Vendor / Third-Party Risk

  • Questionnaires, right-to-audit clauses, SOC 2 reports
  • Supply chain risk — hardware/software compromised before delivery
  • MOU — Memorandum of Understanding; informal agreement
  • MSA — Master Service Agreement; legal contract
  • SLA — Service Level Agreement; defines uptime/support commitments
  • NDA — Non-disclosure Agreement; protects confidential info

Change Management

  • RFC (Request for Change) → CAB approval → Maintenance window → Rollback plan → Post-change review
  • CAB — Change Advisory Board; approves significant changes
  • Emergency change — expedited process for urgent fixes
  • Technical implications: service outages, legacy dependencies, version control
  • Documented rollback plan is required for every change
⚠ Exam Trap
BCP vs DRP: BCP is the broader plan to keep the business running (people, processes, communication). DRP is the IT-specific plan to restore systems. BCP includes DRP. A tabletop exercise tests the plan with discussion — no systems are actually used.

Tip — Security control types: six types, two categories
The six types: Preventive, Detective, Corrective, Deterrent, Compensating, Directive.
The three functional categories: Technical, Administrative (Managerial), Physical.

A single control can have multiple labels — e.g., a security camera is Physical (category) + Detective (type). Exam questions frequently ask for both. Know which type stops vs. which finds.
Trap — Asymmetric encryption direction
Encrypt data → use the recipient's public key (anyone can encrypt; only recipient decrypts with private key).
Sign data / prove identity → use your own private key (anyone verifies with your public key).

The exam will flip these. Remember: public key = lock (anyone can lock), private key = key (only you can unlock).
Trap — RADIUS vs TACACS+
RADIUS: UDP, encrypts password only, combines authentication + authorisation, widely supported.
TACACS+: TCP, encrypts the entire packet, separates authentication / authorisation / accounting into distinct functions, Cisco-centric.

If the question says "encrypts all traffic" or "Cisco network devices" — answer is TACACS+.
Tip — EAP method strength ladder
EAP-TLS (strongest) → mutual certificate auth, both client and server need certs
PEAP → TLS tunnel with only server cert; inner method is MSCHAPv2 (password-based)
EAP-TTLS → similar to PEAP; more flexible inner methods
EAP-MD5 (weakest) → no mutual auth; vulnerable to MITM

"Strongest Wi-Fi authentication" = EAP-TLS (certificate on every device).
Trap — Symmetric vs Asymmetric speed
Asymmetric (RSA, ECC) is much slower than symmetric (AES). TLS solves this with a hybrid approach: asymmetric to securely exchange a symmetric key, then symmetric for all bulk data. The exam may test whether you know that asymmetric alone is impractical for large data volumes.
Tip — Incident Response: PICERL in order
Preparation → Identification → Containment → Eradication → Recovery → Lessons Learned

Common exam trap: placing "Containment" before "Identification." You must identify the scope of the incident before containing it. Also: evidence is collected during Identification and Containment — preserve chain of custody (order of volatility: RAM first, then disk, then logs).
Trap — False Positive vs False Negative
False Positive: IDS alerts when there is NO real threat. Annoying but safe — just noise.
False Negative: IDS misses a REAL threat. This is worse — the attack goes undetected.

Tuning an IDS to reduce false positives may increase false negatives. Exam questions about which is "more dangerous" → false negative.
Tip — Cloud shared responsibility by model
IaaS (e.g., EC2): customer owns OS, middleware, app, data, network config.
PaaS (e.g., Elastic Beanstalk): customer owns app code and data; provider manages runtime/OS.
SaaS (e.g., Microsoft 365): customer manages data, users, and access policies only.

The lower the abstraction level, the more the customer is responsible for.
Trap — Data destruction methods by media type
Degaussing: magnetic media only (HDDs, tapes) — useless on SSDs or optical media.
Overwriting (7-pass): HDDs only — SSDs have wear levelling that prevents reliable overwriting.
Shredding / Pulverising: works on all media; highest assurance.
Cryptographic erasure (crypto-shredding): delete the encryption key — data is unrecoverable; ideal for cloud/SSD.

For SSDs and cloud storage → crypto-shredding or physical destruction.
Tip — SY0-701 scenario keyword shortcuts
  • "Prevent malware from calling home" → DNS sinkhole or proxy with URL filtering
  • "Detect lateral movement / abnormal user behaviour" → UEBA (User and Entity Behaviour Analytics)
  • "Reduce attack surface on servers" → application allowlisting + disable unnecessary services/ports
  • "Strongest encryption for data at rest" → AES-256
  • "Fastest asymmetric for mobile / IoT" → ECC (smaller keys, lower compute)
  • "Non-repudiation" → digital signature (private key signs; proves sender cannot deny)
  • "Integrity only — no confidentiality" → hash (SHA-256) or HMAC
  • "Full AD compromise via one credential" → Golden Ticket attack (KRBTGT hash)
  • "Secure password storage" → bcrypt or Argon2 (NOT SHA-256 alone)
  • "Automatically respond to threats" → SOAR (not SIEM — SIEM only alerts)
Trap — Whaling vs BEC (most-confused pair on exam)
Whaling: the C-suite executive IS the victim — someone is trying to steal the exec's credentials or trick the exec into an action.
BEC (Business Email Compromise): the attacker impersonates a C-suite exec to trick a subordinate (e.g., CFO gets "CEO" email requesting urgent wire transfer).

The direction of deception is the key: whaling targets the exec; BEC uses the exec's identity as a weapon against others.
Tip — Vulnerability scanning vs Penetration testing
Vulnerability scan: automated tool identifies known weaknesses (CVEs, misconfigurations); non-exploitative; can be run frequently.
Penetration test: human tester actively exploits vulnerabilities to determine real-world impact; authorised, scoped engagement; less frequent.

Bug bounty: ongoing, crowdsourced — external researchers report vulns for a reward.
Red team: full adversary simulation — physical, social, and technical attack chains.
