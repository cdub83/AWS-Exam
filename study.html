VPC Fundamentals

CIDR & Subnets

  • VPC size: /16 to /28
  • RFC 1918: 10.x, 172.16–31.x, 192.168.x
  • CGNAT 100.64.0.0/10 valid for VPC
  • 5 IPs reserved per subnet (network, router, DNS, future, broadcast)
  • /28 subnet → 11 usable IPs

Internet Access

  • Internet Gateway — bidirectional IPv4/IPv6
  • NAT Gateway — outbound-only IPv4 from private subnets
  • Egress-Only IGW — outbound-only IPv6 (no NAT)
  • NAT Gateway: one per AZ for HA; scales to 45 Gbps
  • NAT instance: must disable source/dest check

MTU / Jumbo Frames

  • In-VPC (same VPC): 9001 bytes
  • Through Transit Gateway: 8500 bytes
  • Internet Gateway / VPN: 1500 bytes
  • DX private VIF: 9001 bytes
  • Set OS MTU to 9001 for in-VPC jumbo frames
VPC Peering
⚠ Exam Trap
VPC peering is non-transitive — traffic cannot flow through a peered VPC to reach a third VPC. Also requires non-overlapping CIDRs at the VPC level (subnet-level differences do not matter). Always requires route table updates AND Security Group rules in both VPCs.

Peering Rules

  • Non-transitive — no hairpinning
  • No overlapping CIDRs (VPC level)
  • Works cross-account and cross-Region
  • Inter-Region: no SG ID referencing (use CIDR)
  • Traffic NOT encrypted at network layer

Setup Steps

  • 1. Create peering connection & accept
  • 2. Add routes in BOTH VPC route tables
  • 3. Update Security Groups to allow peer CIDR
  • 4. (Optional) Enable DNS resolution
Transit Gateway
✓ Key Concept
TGW is a hub-and-spoke network transit hub. It is transitive — spoke VPCs can reach each other through TGW. Each attachment has its own entry in a TGW route table. Route propagation automatically adds attachment CIDRs.

TGW Routing

  • Multiple route tables for segmentation
  • Route propagation auto-adds VPC CIDRs
  • Blackhole routes block specific CIDRs
  • Appliance Mode: symmetric routing for stateful firewalls
  • ECMP: requires BGP + enabled on TGW

TGW Attachment Types

  • VPC — up to 5 per VPC
  • VPN — Site-to-Site VPN
  • Direct Connect Gateway — via transit VIF
  • Peering — inter-Region TGW (AES-256 encrypted)
  • Connect — GRE+BGP for SD-WAN

TGW Special Features

  • Multicast: must enable at creation time
  • Connect: SD-WAN via GRE tunnel, BGP, up to 5 Gbps
  • Inter-region peering: auto-encrypted AES-256-GCM
  • Flow Logs: captures TGW-level drops invisible to VPC Flow Logs
  • Network Manager: global topology + Route Analyzer
AWS PrivateLink

Interface Endpoints

  • ENI with private IP in your VPC subnet
  • Private DNS: standard hostname → endpoint IP
  • Endpoint policy restricts API actions
  • Works with overlapping CIDRs
  • Charged per AZ + per GB

Gateway Endpoints

  • S3 and DynamoDB only
  • Added to route table as a prefix list target
  • Free — no per-hour or per-GB cost
  • Does not use ENI — no private IP
  • Cannot be used across VPC peering/TGW

Endpoint Services (PrivateLink)

  • Expose service behind NLB to other VPCs
  • Works with overlapping consumer CIDRs
  • Consumer creates Interface Endpoint to connect
  • Allowed principals: control who can connect
  • Manual acceptance optional

VIF Types

  • Private VIF → VGW (one VPC) or DXGW
  • Public VIF → AWS public IP space (S3, DynamoDB public endpoints) — NOT internet
  • Transit VIF → DXGW only (connects to TGW)
  • Hosted connections: only one VIF per connection

DX Gateway

  • Associates private or transit VIFs to VGWs/TGWs
  • Up to 10 VGWs or 6 TGWs
  • Cross-account TGW association via RAM
  • Associated TGW CIDRs must be non-overlapping
  • Default AWS BGP ASN: 7224

Resiliency Models

  • Maximum: 2 locations × 2 devices × 2 connections
  • High: 2 locations, 1 connection each
  • Development: 1 location, 1 connection + VPN backup
  • BFD: 300 ms interval × 3 = ~900 ms detection
  • LAG: up to 4 active connections, same location
⚠ Exam Trap
Transitive routing does NOT work through VPC peering over Direct Connect. On-premises → DX → VPC-A → peering → VPC-B is not supported. Use Transit Gateway with a transit VIF instead.

Route 53 Resolver

  • Inbound Endpoint — on-premises resolvers → VPC (resolve private zones)
  • Outbound Endpoint — VPC → on-premises DNS (forwarding rules)
  • Share forwarding rules via RAM across org
  • DNS Firewall: block/alert on domain queries (AWS managed threat lists)

ALIAS vs CNAME

  • ALIAS works at zone apex; CNAME does not
  • ALIAS → ALB/NLB/CloudFront/S3 website: no extra query charge
  • CNAME cannot coexist with other records at same name
  • ALIAS returns A records to clients

Health Checks

  • Endpoint, CloudWatch Alarm, or Calculated
  • Calculated: aggregate up to 256 child checks
  • Failure threshold: 1–10 consecutive failures
  • DNSSEC: protects against cache poisoning/spoofing
CloudFront

Origin Access

  • OAC (Origin Access Control) — current recommended method for S3
  • OAI (Origin Access Identity) — legacy, still valid on exam
  • Custom header → restrict ALB to CloudFront only
  • CloudFront managed prefix list: secondary control

Edge Compute

  • CloudFront Functions: <1 ms, all PoPs, simple JS, header/URL manipulation
  • Lambda@Edge: 1–5 ms, Node.js/Python, network calls, complex logic
  • CF Functions: viewer request/response events only
  • Lambda@Edge: all 4 events (viewer req/res, origin req/res)

High Availability

  • Origin Groups: primary + failover on 5xx or connection fail
  • Origin Shield: extra caching layer → reduces origin load
  • Signed URLs (custom policy): expiration + IP restriction
  • Signed cookies: multi-file access for authenticated users
  • Geo restriction: block/allow by country
Site-to-Site VPN

VPN Basics

  • 2 tunnels per connection (active/active capable)
  • Up to 1.25 Gbps per tunnel
  • IKEv1 and IKEv2 supported (per tunnel)
  • IKEv2: fewer round trips, built-in NAT-T
  • Accelerated VPN: routes via Global Accelerator edge

VPN + TGW ECMP

  • Enable ECMP on TGW
  • Customer GW advertises same BGP routes over both tunnels
  • TGW distributes flows across tunnels
  • Doubles effective throughput (~2.5 Gbps)
  • Requires BGP — not available with static routing

VPC Flow Logs
Action valueMeaning
ACCEPTPacket permitted by Security Group and/or NACL — does NOT confirm TCP handshake or app response
REJECTPacket denied by Security Group (implicit deny) or NACL explicit DENY rule
NODATANo traffic to/from the ENI during the aggregation window — not an error
SKIPDATALogging system internal capacity issue — some flow records were dropped

Flow Log Levels

  • VPC level → all ENIs in all subnets
  • Subnet level → all ENIs in that subnet
  • ENI level → specific interface only
  • Default aggregation: 10 minutes; optional: 1 minute

Custom Fields

  • pkt-srcaddr — original source IP (pre-NAT)
  • pkt-dstaddr — original dest IP (pre-NAT)
  • srcaddr/dstaddr show post-NAT IPs when NAT Gateway is in path
  • Deliver to S3, CloudWatch Logs, or Kinesis Data Firehose
⚠ Exam Trap
ACCEPT in Flow Logs only confirms the network-layer security decision. It does not mean the TCP connection was established or data was transferred successfully. Application-layer issues (app crash, TLS mismatch) won't appear in Flow Logs.

Key ENA Metrics

  • bw_in/out_allowance_exceeded — bandwidth limit drops
  • pps_in/out_allowance_exceeded — packet rate limit drops
  • conntrack_allowance_exceeded — connection tracking table full
  • linklocal_allowance_exceeded — metadata/DNS traffic throttled
  • ENA Express (SRD): improves tail latency (p99), not max bandwidth

CloudTrail vs Config

  • CloudTrail: WHO made what API call, WHEN, from WHERE (identity, source IP, timestamp)
  • AWS Config: WHAT changed (before/after configuration state), continuous compliance
  • Use CloudTrail to find who changed a Security Group
  • Use Config to detect/remediate non-compliant SG rules

NAT Gateway Timeouts

  • TCP idle timeout: 350 seconds
  • UDP idle timeout: 30 seconds
  • Idle connections dropped after timeout → implement TCP keepalives
  • 55,000 simultaneous connections per unique destination (IP:port)

⚠ Exam Trap
NACLs are stateless. Allowing inbound TCP 443 is not enough — you must also allow outbound ephemeral ports (1024–65535) for the server's response packets to leave the subnet. Security Groups handle this automatically.
AWS WAF

WAF Scope

  • CLOUDFRONT scope: global, must create in us-east-1; attaches to CloudFront only
  • REGIONAL scope: created per Region; attaches to ALB, API Gateway, AppSync, Cognito
  • Cannot share a web ACL between CloudFront and a regional resource

Rule Types

  • Rate-based rules: 1, 2, 5, or 10 minute window; per IP / forwarded IP / custom key
  • Geo match: block/allow by country
  • Managed rule groups: AWS or Marketplace (OWASP, SQLi, XSS, bot control)
  • Custom rules: IP sets, regex, label matching

WAF vs Network Firewall

  • WAF: L7 HTTP/HTTPS inspection for web apps (CloudFront, ALB, API GW)
  • Network Firewall: VPC-level L3/L4/L7, stateful, IPS (Suricata), domain lists
  • Network Firewall: deployed in VPC subnets; traffic must route through it
  • Network Firewall: stateless rules first, then stateful rules

Network Firewall — Rule Processing

Rule Processing Order

  • 1. Stateless rules — per-packet, no state; PASS/DROP/FORWARD
  • 2. Stateful rules — connection-aware; flow-based inspection
  • Stateless DROP never reaches stateful engine
  • Stateless PASS goes to default stateless action
  • FORWARD sends to stateful engine

Stateful Rule Types

  • 5-tuple: src/dst IP, src/dst port, protocol
  • Domain list: SNI inspection + DNS query; allow/deny FQDNs
  • Suricata IPS: full Suricata rule syntax; header content matching
  • Domain lists use SNI for HTTPS, DNS for HTTP
Governance at Scale

Service Control Policies (SCPs)

  • Applied at AWS Organizations level
  • Cannot be overridden by IAM policies in member accounts
  • Preventive control (not detective)
  • Example: deny ec2:AttachInternetGateway, deny ec2:RunInstances with public IP

AWS Firewall Manager

  • Centrally manage: WAF web ACLs, SG policies, Network Firewall, Shield Advanced, DNS Firewall
  • Automatically applies to new accounts/resources in org scope
  • Requires AWS Organizations + Config enabled

AWS Config vs CloudTrail

  • Config: detective — what is the current/past config state?
  • CloudTrail: audit — who made which API call, when?
  • Config conformance packs: deploy rules org-wide via StackSets
  • Config aggregator: multi-account, multi-Region compliance view
VPC Endpoint Security Pattern
✓ Defense in Depth
To restrict S3 access to VPC-only: (1) Create a VPC Gateway Endpoint for S3 to route S3 traffic through the endpoint. (2) Add an S3 bucket policy with aws:SourceVpc condition denying access from outside the VPC. Both controls together provide defense-in-depth — the endpoint policy controls what actions are allowed through the endpoint, and the bucket policy blocks the public endpoint.
